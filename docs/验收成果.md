# 赤峰市安定医院信息化建设项目 — 验收成果清单

文档目的：基于甲方要求（`合同.md`、`技术要求.md`）、乙方投标响应（`技术响应表.md`）和实际交付的技术文档（`chifeng-ai-diagnosis_wiki.md`、`kotaemon_wiki.md`、`parse_data_wiki.md`）整理验收要点、对应验证方法与证据清单，便于甲方组织验收。

注意：本文档以交付的文档与技术响应为依据列出可直接验证的项，同时标注需乙方补交以完成合同性验收的资料。

**验收总览**
- **项目名称**: 赤峰市安定医院信息化建设项目
- **乙方**: 北京远谋科技有限公司
- **交付主体代码/文档**: `chifeng-ai-diagnosis`, `kotaemon`, `parse_data` 及其对应wiki（位于 `docs/` 目录下）

**依据条款与对应验收项**

**功能完整性**
- **要求**: 系统功能需100%覆盖附件一所列功能（病例数据录入、统计分析、科研报告生成等）。
- **验收方法**: 按附件一功能清单逐项测试、逐条通过并记录缺陷率为0。执行端到端演示（样例病历）并比对输出报告结构和内容。
- **可用证据**: `parse_data_wiki.md`（数据解析与报告生成说明）、`chifeng-ai-diagnosis_wiki.md`（系统功能与报告结构）、Streamlit / `app.py`（演示入口）和 `utils/04_generate_reports_infini.py`（报告生成脚本）。
- **当前状态**: 文档与脚本展示了完整的数据处理与报告生成功能；需要提供具体功能测试记录作为验收证据（见需补交项）。

**性能指标**
- **要求**: 单用户核心操作平均响应时间≤5秒；支持≥10并发科研用户；批量导入10条医学样本≤3分钟；数据解析准确率≥99.9%。
- **验收方法**: 在招标指定硬件上进行压力测试和性能测试（TTFT、并发测试、批量导入计时、解析准确率抽样比对）。
- **可用证据**: `技术响应表.md`中测试结果（TTFT=3.8s，支持12路并发，批量处理指标说明）；`parse_data_wiki.md` 与 `service_conf/parse_data/conf/*`（解析配置）。
- **当前状态**: 性能指标在投标响应中声明已满足（TTFT 3.8s、并发12、检索/召回指标通过内部测试）。验收时需在目标环境复现并提交性能测试报告（见需补交项）。

**数据安全与合规**
- **要求**: 患者隐私加密存储与传输，满足个人信息保护法与GB/T39725-2020；三级权限控制；接口需通过伦理委员会审核。
- **验收方法**: 检查加密算法实现、传输通道（TLS/HTTPS/内网隔离）、权限配置和审计日志；伦理合规材料审阅。
- **可用证据**: `技术要求.md`（合规条款），`技术响应表.md`（网络与离线部署说明），部署文档（如 `DOCKER.md`）、网络部署说明在wiki/README或运维手册中。
- **当前状态**: 文档说明了支持双网卡隔离、离线部署和监控预警，但未在现有wiki中找到加密实现细节与权限配置导出文件，需补交加密实现说明、权限矩阵和合规审核记录。

**兼容性与稳定性**
- **要求**: 支持Windows10/11、macOS12+、Chrome/Edge 90+；连续72小时满负荷运行无崩溃；系统日志无ERROR级别；可用性≥99.9%。
- **验收方法**: 在规定平台上进行兼容性测试；执行72小时稳定性运行并提交运行日志与可用性统计；检查系统日志。
- **可用证据**: `技术要求.md`、`技术响应表.md`（稳定性测试结果99.925%），wiki中部署与监控模块说明、`DOCKER.md`。
- **当前状态**: 投标响应提供了99.925%可用性测试数据；需提供72小时稳定性运行日志与环境配置以核验。

**文档完整性与交付物**
- **要求**: 提供可运行的软件安装包（含版本号）与完整源代码（注释率≥80%）；《软件测试报告》（含500条测试用例且全部通过）；《用户操作手册》、《数据安全维护手册》。
- **验收方法**: 检查交付代码、注释比例、验证安装包可用性并运行单元/集成测试，审阅并抽测500条测试用例通过情况，评估用户手册与维护手册的完备性。
- **可用证据**: 现有wiki文件（包含设计与使用说明）、`chifeng-ai-diagnosis_wiki.md`、`kotaemon_wiki.md`、`parse_data_wiki.md`、`DOCKER.md`、`requirements*.txt`。
- **当前状态**: 基本技术文档与部署说明存在，但缺失正式的安装包、完整测试报告（500条用例）和明确的注释率统计，需补交：
  - `软件安装包`（含版本号与安装说明）
  - `测试报告_验收版.pdf`（含500条测试用例和执行记录）
  - `源代码压缩包` 或 提供 `git` 访问并告知提交tag/commit
  - `注释覆盖率报告`（注释率≥80%证明）

**售后服务与现场支持**
- **要求**: 质量保修1年，乙方应提供维护手册、备件清单、维修密码等；乙方需在需求确认、部署、验收阶段累计现场服务≥14天。
- **验收方法**: 检查保修协议文本、维护手册、现场服务记录（签到、出差单、会议纪要）。
- **可用证据**: `合同.md`、`技术要求.md`。
- **当前状态**: 合同条款明确，需补交现场服务计划与已完成的现场服务签到记录（若尚未完成则提交计划）。

**模型与检索性能（RAG/Embedding/Rerank等）**
- **要求**: 嵌入模型、检索召回与重排序满足投标要求（详见`技术响应表.md`）。
- **验收方法**: 用内部测试集复现召回、nDCG/MRR、Recall@5等指标，检验嵌入上下文长度、检索K/N配置项、生效性。
- **可用证据**: `技术响应表.md`（Qwen3-Embedding-8B、35K token 支持、Recall@5=88.2%）、`kotaemon_wiki.md`（向量索引与嵌入说明）、`service_conf/parse_data/conf/llm.json` 或 wiki 中模型配置引用。
- **当前状态**: 投标响应给出具体数值并声明已通过内部测试；建议在目标环境复现并提交测试脚本与结果数据文件（CSV/JSON）作为验收证据。

**验收交付清单（需提交的交付物）**
1. 可运行软件安装包（含版本号）或可用 `Docker` 镜像与 `Dockerfile`、启动脚本。

4. 性能测试报告：包含TTFT、并发压力测试、批量导入计时、72小时稳定性运行日志与可用性统计。

8. 注释覆盖率或代码质量报告、依赖清单（`requirements.txt`/`requirements_macos.txt`）。
